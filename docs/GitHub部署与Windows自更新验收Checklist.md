# GitHub部署与Windows自更新验收Checklist

## 1. 使用目的

本 Checklist 用于验收：

- 项目是否具备 GitHub 代码托管能力
- Windows 目标机是否可稳定部署
- 后续是否可自行更新
- 更新失败时是否可回滚

说明：

- 本清单强调“代码更新与数据隔离”
- 不允许只验证“项目能启动”就判定通过

---

## 2. 仓库与目录验收

### 2.1 GitHub 仓库

- [ ] 项目代码已放入 GitHub 仓库
- [ ] 真实数据库未提交到 GitHub
- [ ] `.env` 未直接提交生产敏感内容
- [ ] 备份文件未提交到 GitHub

### 2.2 Windows 目录结构

- [ ] 已确定 Windows 根目录
- [ ] 代码目录与数据目录分离
- [ ] 备份目录独立
- [ ] 日志目录独立

推荐结构示例：

- [ ] `app/`
- [ ] `data/`
- [ ] `backups/`
- [ ] `logs/`

---

## 3. 路径配置化验收

- [ ] 数据库主路径可通过环境变量配置
- [ ] session 数据路径可通过环境变量配置
- [ ] 初始化脚本支持配置化路径
- [ ] 更新脚本不会写死当前开发机路径
- [ ] 目标机 `.env` 中可独立指定数据库位置

---

## 4. 启动能力验收

- [ ] 在 Windows 目标机安装 Node.js LTS
- [ ] 在 Windows 目标机安装 Git
- [ ] 已成功执行依赖安装
- [ ] 服务可正常启动
- [ ] 首页可访问
- [ ] `/api/health` 返回正常

---

## 5. 更新能力验收

- [ ] 更新脚本存在
- [ ] 更新脚本可执行
- [ ] 更新前会自动备份数据库
- [ ] 更新时会拉取 GitHub 最新代码或指定版本
- [ ] 更新后会安装依赖
- [ ] 更新后服务能重新启动
- [ ] 更新后 `/api/health` 正常

---

## 6. 数据安全验收

- [ ] 更新前数据库已备份
- [ ] 更新后真实业务数据仍存在
- [ ] 更新后登录仍正常
- [ ] 更新后物料、BOM、SOP、工单数据未丢失
- [ ] 更新不会覆盖 `data/` 目录

---

## 7. 回滚能力验收

- [ ] 回滚脚本存在
- [ ] 回滚脚本可执行
- [ ] 能恢复到上一个代码版本
- [ ] 能恢复更新前数据库备份
- [ ] 回滚后服务可启动
- [ ] 回滚后 `/api/health` 正常

---

## 8. 实机演练验收

### 8.1 首次部署演练

- [ ] 在目标机完成首次部署
- [ ] 可登录系统
- [ ] 页面可正常访问

### 8.2 更新演练

- [ ] 完成至少一次真实更新演练
- [ ] 更新后核心页面可正常打开
- [ ] 更新后数据库无异常

### 8.3 回滚演练

- [ ] 完成至少一次真实回滚演练
- [ ] 回滚后系统恢复正常

---

## 9. 风险验收

- [ ] 未再依赖开发机绝对路径
- [ ] 未把真实生产数据库交给 Git 覆盖
- [ ] 未把更新能力建立在手工拷贝目录之上
- [ ] 未缺失健康检查步骤
- [ ] 未缺失回滚路径

---

## 10. 最终通过标准

全部满足以下条件时，才可判定通过：

- [ ] Windows 目标机能独立运行系统
- [ ] GitHub 能作为后续代码更新源
- [ ] 更新不会覆盖真实数据
- [ ] 更新脚本可用
- [ ] 回滚脚本可用
- [ ] 已完成真实更新演练
- [ ] 已完成真实回滚演练
