# 当前项目技术说明与维护交接指南

## 1. 文档目的

本文档用于把当前项目完整交接给其他同事，目标是让接手人满足以下条件：

- 不使用你当前这台电脑
- 不使用你当前的 Codex 账号
- 只拿到项目目录、数据库和必要配置后，就能独立启动、维护和继续开发

本文档不是产品说明书，而是技术交接与维护手册。

## 1.1 名称约定

当前项目存在两个并行名称，后续维护时请明确区分：

- 仓库名：`OvO-Inventory-System`
  - 用于 GitHub、tag、release、更新脚本
- 本地目录名：`inventory-system-codex`
  - 用于历史开发过程中的常见本地目录名
  - 这是目录名示例，不要求每台机器完全一致

这不是两个项目，而是同一个项目在“仓库层”和“本地目录层”的不同命名。

## 2. 项目定位

本项目是一个面向内部生产型企业的物料、库存、生产、发货一体化系统。本文档后续统一使用：

- `<repo-root>` 表示代码仓库根目录
- `<data-dir>` 表示数据库目录

当前设计目标不是通用电商库存系统，而是服务于以下业务场景：

- 多仓库存管理
- 生产型物料主数据管理
- BOM / SOP / 工单驱动的领料、完工、退料
- 销售发货与库存联动
- `收 / 发 / 转 / 盘` 的执行型作业入口
- `进销存` 与 `动作流水` 并存的双层统计
- 工单异常、红冲、替代料、双预警等生产型扩展能力

当前系统已经从“后台原型”推进到了“可持续迭代的正式业务骨架”阶段。

## 4. 当前运行方式

### 4.1 技术栈

- 后端：`Node.js + Express`
- 数据库：`SQLite (better-sqlite3)`
- Session：`express-session + connect-sqlite3`
- 前端：单文件 SPA，主文件为 `/public/index.html`
- Excel 导入：`exceljs`
- 测试：`vitest + supertest`

### 4.2 运行入口

项目主入口：

- `server/index.js`

常用命令：

```bash
npm install
npm start
npm run dev
npm run init-db
npm run import:inventory-costs
npm test
```

当前默认访问地址：

- [http://localhost:3000](http://localhost:3000)

### 4.3 环境变量

当前 `.env` 内容结构很简单：

```env
PORT=3000
SESSION_SECRET=...
NODE_ENV=development
```

字段含义：

- `PORT`
  - 服务端口
- `SESSION_SECRET`
  - Session 签名密钥
  - 迁移到新电脑或正式环境时必须改成新的随机字符串
- `NODE_ENV`
  - 当前一般用 `development`
  - 正式环境建议改成 `production`

### 4.4 健康检查

健康检查接口：

- [http://localhost:3000/api/health](http://localhost:3000/api/health)

建议每次交接、迁移、升级后第一步先检查这个接口。

- 新迁移必须可重复执行且幂等

## 10. 当前已实现的重要增强

### 10.1 双预警模型

系统当前有两套预警：

- 物料级预警
  - `safety_stock` 优先，缺省回退 `min_stock`
- 套数级预警
  - 基于有效 BOM 计算可生产套数
  - 小于目标保供套数时告警

### 10.2 供给方式模型

不是只支持线缆，而是通用物料供给策略：

- `purchase_only`
- `direct_issue`
- `prebuild_wip`
- `on_site_fabrication`

### 10.3 替代料执行模型

已具备：

- BOM / SOP 上声明允许替代与优先级
- 工单缺料时显示替代候选
- 工单替代方案维护
- 领料时实际使用替代料
- 工单快照记录实际替代
- 单据明细、台账、审计可追溯原料与替代原因

### 10.4 统一物料搜索选择器

目前系统已经在多个高频入口使用统一搜索式选料组件，而不再依赖长下拉滚动：

- 收 / 发 / 转 / 盘执行页
- 发货单
- BOM
- 生产工单
- 生产异常单
- 替代料选择

已经具备：

- 名称 / 编码 / 拼音 / 规格搜索
- 最近使用
- 场景推荐
- 同名物料高风险提示
- 停用 / 冻结 / 低库存 / 负库存状态标签
- 批量粘贴带料
- 批量粘贴失败明细预览

## 19. 同事继续开发时的工作约定

如果后续由其他同事继续开发，建议统一遵守以下约定：

### 19.1 后端改动

- 路由层放在 `server/routes`
- 复杂库存/单据逻辑优先放在 `server/services`
- 新增结构改动一律走 `migrations.js`

### 19.2 前端改动

- 当前前端仍以 `/public/index.html` 为主
- 页面组件、状态和弹窗都在同一个文件里
- 维护时先搜索现有组件复用，不要随意再造一套交互

### 19.3 数据改动

- 导入脚本放 `deploy`
- 不要直接对 `inventory.db` 批量手写 SQL 改业务数据，除非先备份并留操作记录
