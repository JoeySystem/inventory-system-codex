# 收发转盘与进销存双层架构实施方案

## 1. 文档目的

本文档用于定义 `inventory-system-codex` 项目在仓储与生产业务上的“双层语言体系”改造方案：

- UI/UX 与现场作业层使用 `收 / 发 / 转 / 盘` 作为一级业务模块
- 数据库核算与报表层继续使用 `期初 + 入 - 出 = 期末` 的进销存平衡口径

本文档面向：

- 产品设计
- 前后端开发
- 数据建模与报表设计
- 测试与上线实施
- 管理层业务确认

## 2. 背景与适用性评估

### 2.1 公司业务背景

当前系统主要服务于公司内部使用，业务特征包括：

- 存在车间生产环节
- 生产对象为射频设备、无人机反制设备及反反制设备
- 同时涉及原材料、半成品、成品、辅料、返修件、报废件等复杂库存对象
- 库存流动既包含贸易型动作，也包含生产型动作

### 2.2 对建议的客观评估

建议内容：

- 在 UI/UX 层使用 `收、发、转、盘` 作为一级功能模块
- 在数据库逻辑与报表层保留 `进销存` 平衡公式

结论：`合理，且适合当前公司场景`

原因如下：

- 对仓库管理员和车间主管来说，`收 / 发 / 转 / 盘` 是天然的作业语言
- 对财务、老板和经营分析来说，`期初 + 入 - 出 = 期末` 是天然的核算语言
- 生产型企业的现场动作远比贸易型企业复杂，直接用“采购/销售/库存”做前台入口会增加认知负担
- 只要底层保留明确业务来源类型，动作语言与核算语言可以并存而不冲突

### 2.3 适用前提

本方案成立的前提是：

- 前台入口与后台核算口径分层设计
- 每一笔库存变动都保留明确 `业务来源类型`
- 报表层不是直接按页面模块名称汇总，而是按标准核算口径汇总

如果只改页面名称，不保留底层来源类型，本方案会导致：

- 报表口径混乱
- 无法区分采购入库与完工入库
- 无法区分销售出库与生产领料
- 财务与经营分析无法落地

## 3. 总体设计原则

本次改造采用以下四条原则：

### 3.1 前台动作语言原则

一级入口按现场动作组织：

- 收
- 发
- 转
- 盘

这样能降低一线人员操作负担。

### 3.2 后台核算语言原则

库存与金额核算仍使用统一公式：

- 期初
- 本期入
- 本期出
- 期末

这样能保证报表稳定和财务可解释性。

### 3.3 业务来源显式化原则

所有库存变动都必须记录：

- 动作类型
- 业务来源类型
- 来源单据

不能只记录“收了一笔”或“发了一笔”。

### 3.4 单据驱动库存原则

库存变化必须由业务单据驱动，不允许随意改余额。

允许影响库存的对象应包括：

- 收货单
- 发料/发货单
- 调拨单
- 盘点/调整单
- 生产领料与完工单

## 4. 目标业务架构

### 4.1 一级功能模块

建议主导航调整为：

- 仪表盘
- 物料
- BOM
- SOP
- 收
- 发
- 转
- 盘
- 生产
- 报表
- 系统设置

### 4.2 一级模块定义

#### 收

用于表达库存增加的现场动作。

包含但不限于：

- 采购收货入库
- 生产完工入库
- 退料回库
- 客退回库
- 外协回厂入库
- 其他入库

#### 发

用于表达库存减少的现场动作。

包含但不限于：

- 生产领料
- 销售发货
- 样机借出
- 研发试制领料
- 报废出库
- 其他出库

#### 转

用于表达库存位置变化但总量不变的现场动作。

包含但不限于：

- 主仓到车间仓
- 车间仓到线边仓
- 线边仓退回主仓
- 仓间调拨

#### 盘

用于表达库存核对与差异调整。

包含但不限于：

- 盘点单
- 盘盈调整
- 盘亏调整
- 报废调整
- 呆滞料清理调整

## 5. 双层模型设计

### 5.1 三层表达结构

系统需要同时存在三层表达：

1. 页面动作层
   - 收 / 发 / 转 / 盘
2. 业务来源层
   - 采购收货 / 完工入库 / 生产领料 / 销售发货 / 调拨 / 盘盈盘亏
3. 核算汇总层
   - 期初 / 入 / 出 / 期末

### 5.2 映射关系

| 页面动作 | 业务来源类型 | 核算归类 |
| --- | --- | --- |
| 收 | purchase_receipt | 入 |
| 收 | production_receipt | 入 |
| 收 | return_to_stock | 入 |
| 收 | customer_return | 入 |
| 发 | production_issue | 出 |
| 发 | sales_shipment | 出 |
| 发 | sample_issue | 出 |
| 发 | scrap_issue | 出 |
| 转 | warehouse_transfer | 不计入入/出汇总，仅位置变化 |
| 盘 | count_gain | 入 |
| 盘 | count_loss | 出 |
| 盘 | manual_adjust | 视调整方向计入入或出 |

结论：

- 页面动作层服务于使用者
- 核算归类层服务于报表
- `业务来源类型` 是两者之间的桥梁

## 6. 数据模型方案

### 6.1 库存流水核心字段

建议在现有库存流水模型中统一补齐以下字段：

- `movement_type`
  - 枚举：`in`, `out`, `transfer`, `count_adjust`
- `biz_type`
  - 枚举建议：
    - `purchase_receipt`
    - `production_receipt`
    - `return_to_stock`
    - `customer_return`
    - `production_issue`
    - `sales_shipment`
    - `sample_issue`
    - `scrap_issue`
    - `warehouse_transfer`
    - `count_gain`
    - `count_loss`
    - `manual_adjust`
- `source_doc_type`
- `source_doc_id`
- `warehouse_id`
- `to_warehouse_id`
- `material_id`
- `quantity`
- `unit_cost`
- `amount`
- `occurred_at`
- `created_by`

### 6.2 单据主表建议

建议后续将动作入口对应到标准单据：

- `receipts`
- `issues`
- `transfers`
- `stock_counts`
- `stock_adjustments`

如短期不拆表，至少要保证现有单据模型拥有：

- 单据编号
- 单据类型
- 业务来源类型
- 状态机
- 审核人/操作人
- 关联库存流水

### 6.3 状态机原则

所有会影响库存的单据需要明确状态机：

- 草稿
- 待确认/待审核
- 已执行
- 已取消

库存变化只允许在规定状态发生，不能出现：

- 建单即扣库存但状态仍是待确认
- 取消时无规则回退
- 重复提交导致重复入库/出库

- 半成品完工入库
- 成品完工入库
- 车间退料回库

#### 发

- 样机借出
- 报废出库

#### 转

- 线边仓退回

#### 盘

## 9. 报表方案

### 9.1 保留进销存主报表

老板和财务侧主报表仍应保留：

- 库存余额表
- 进销存汇总表
- 库存价值表
- 周转分析表

核心口径：

- 期初数量/金额
- 本期入数量/金额
- 本期出数量/金额
- 期末数量/金额

### 9.2 增加动作报表

给仓库与车间主管的报表建议新增：

- 今日收货汇总
- 今日发料汇总
- 今日调拨汇总
- 今日盘点差异

### 9.3 增加来源结构报表

给管理层建议新增：

- 入库来源结构分析
  - 采购入库
  - 完工入库
  - 退料回库
  - 客退回库
- 出库来源结构分析
  - 销售发货
  - 生产领料
  - 报废出库
  - 样机借出

### 9.4 转和盘的报表原则

- `转` 不计入经营层面的入/出总量，但保留仓位变化统计
- `盘` 不作为日常经营动作展示，但需要进入差异分析和损耗分析

## 10. 权限模型建议

建议按动作职责划分权限，而不是按“进销存模块”粗粒度划分。

### 10.1 角色建议

- 仓库员
  - 收、发、转、盘录入
- 仓库主管
  - 审核收发转盘单据
  - 查看动作报表
- 车间主管
  - 发料申请
  - 完工入库确认
  - 车间转仓申请
- PMC/生产计划
  - 工单、齐套分析、领料计划
- 财务/老板
  - 进销存、库存价值、差异分析、经营报表
- 系统管理员
  - 全部权限

### 10.2 权限粒度建议

建议至少区分：

- 查看
- 新建
- 编辑
- 审核/执行
- 取消
- 导出

## 11. 当前系统改造路径

### 11.1 第一阶段：信息架构调整

目标：

- 先把前台导航和入口改造成 `收 / 发 / 转 / 盘`
- 不立即大改底层表结构

内容：

- 菜单重构
- 页面入口重组
- 旧功能重新归类
- 用户手册同步调整

### 11.2 第二阶段：业务类型补齐

目标：

- 给现有库存流水和相关单据增加 `biz_type`
- 明确来源类型

内容：

- 数据库字段扩展
- 接口契约调整
- 单据写入逻辑改造
- 历史数据回填

### 11.3 第三阶段：报表重构

目标：

- 建立动作报表、来源结构报表与进销存报表三套视图

内容：

- 仓库动作看板
- 库存核算报表
- 差异与损耗分析

### 11.4 第四阶段：生产联动深化

目标：

- 将 `发` 与 `生产领料`
- 将 `收` 与 `完工入库`
- 将 `盘` 与 `报废/返修`

全部打通到生产流程中。

## 12. 口径定义要求

本项目必须形成正式口径文档，至少明确以下问题：

- 哪些业务算“入”
- 哪些业务算“出”
- 哪些业务只算“转”
- 盘盈盘亏如何进入报表
- 报废如何计入出库或损耗
- 完工入库是否计入经营入库
- 客退回库是否单列

建议后续将该口径文档与系统配置绑定，避免：

- 页面叫法和报表叫法不一致
- 不同部门口径不同
- 测试与正式统计结果不一致

## 13. 风险与控制点

### 13.1 主要风险

- 只改菜单名称，不改底层映射
- 动作入口过于简化，掩盖真实来源类型
- 转和盘处理不清，导致报表扭曲
- 历史数据没有 `biz_type`，迁移后统计失真

### 13.2 控制措施

- 所有库存流水强制带 `biz_type`
- 所有报表统一由核算层规则生成
- 前台动作页必须可见来源类型
- 导入历史数据时补齐映射规则
- 上线前做逐单回放校验

## 14. 验收标准

本方案完成后，应满足以下验收标准：

- 仓库人员能在 3 次点击内进入 `收 / 发 / 转 / 盘` 常用场景
- 任何库存流水都能追到具体来源类型与来源单据
- 报表层仍可输出标准进销存结果
- `转` 不会被错误记入经营入/出
- `盘盈盘亏` 能正确进入差异与损耗分析
- 生产领料和完工入库可分别进入 `发` 与 `收`
- 财务与老板看到的库存余额与价值口径稳定可解释

## 15. 结论

针对当前公司“自用、带车间、带生产、带射频设备制造”的业务背景，本方案结论如下：

- 在 UI/UX 层使用 `收 / 发 / 转 / 盘` 作为一级功能模块是合理的
- 在数据库与报表层继续使用 `进销存平衡公式` 也是必须保留的
- 正确做法不是二选一，而是通过 `业务来源类型` 建立两套语言之间的桥梁

最终目标应为：

- 现场人员用动作语言高效操作
- 管理层与财务用核算语言稳定分析
- 系统底层用统一流水与口径规则保证一致性
