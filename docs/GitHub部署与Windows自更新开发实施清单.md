# GitHub部署与Windows自更新开发实施清单

## 1. 文档目的

本文档用于把 [GitHub部署与Windows自更新方案.md](./GitHub部署与Windows自更新方案.md) 拆成可执行的开发任务清单，便于后续逐项落地、分工和跟踪。

原则：

- 先做结构准备
- 再做代码适配
- 再做更新机制
- 最后做实机验证

---

## 2. 实施范围

本次实施范围包括：

1. GitHub 代码管理方案落地
2. Windows 部署目录规范落地
3. 数据目录与代码目录解耦
4. 更新脚本设计与实现
5. 回滚和健康检查机制
6. 目标 Windows 机器更新演练

不在本轮范围内：

- 数据库更换为 PostgreSQL
- 自动在线更新客户端
- 多机集中部署平台

---

## 3. 阶段拆分

### 阶段 A：仓库与目录规范

目标：

- 明确 GitHub 代码仓库用途
- 明确 Windows 目标机目录结构
- 明确哪些文件跟 Git，哪些不跟 Git

任务：

1. 确认 GitHub 仓库作为唯一代码源
2. 明确 Windows 根目录，例如 `D:\\OvO-System`
3. 明确目录结构：
   - `app`
   - `data`
   - `backups`
   - `logs`
4. 确认 `.gitignore` 不提交运行数据与备份
5. 明确 `.env` 放置策略

交付物：

- 最终目录结构说明
- GitHub 使用约定

---

### 阶段 B：代码适配

目标：

- 让数据库、session、日志路径不再强依赖代码目录

任务：

1. 改造数据库主路径为环境变量可配置
   - `DB_PATH`
2. 改造 session 存储目录为环境变量可配置
   - `SESSION_DB_DIR`
3. 检查数据库初始化脚本是否也支持外部路径
4. 检查导入/维护脚本是否仍写死项目内路径
5. 更新 `.env` 示例模板

重点文件：

- `server/db/database.js`
- `server/index.js`
- `server/db/init.js`
- 相关 deploy 脚本

交付物：

- 代码目录与数据目录解耦
- 可配置路径说明

---

### 阶段 C：更新机制

目标：

- 让 Windows 机器可按统一脚本完成更新

任务：

1. 新增 `deploy/update.bat`
   - 拉代码
   - 安装依赖
   - 跑迁移
2. 新增 `deploy/update-and-restart.bat`
   - 停服务
   - 备份数据库
   - 更新
   - 启动服务
   - 健康检查
3. 为更新脚本增加清晰日志输出
4. 明确更新脚本支持的模式：
   - 指定 tag
   - 当前分支
5. 加入失败中止逻辑

交付物：

- 更新脚本
- 更新操作说明

---

### 阶段 D：回滚机制

目标：

- 更新失败时能够恢复上一个可用版本

任务：

1. 新增 `deploy/rollback.bat`
2. 明确回滚依赖：
   - Git 版本号
   - 更新前数据库备份
3. 回滚后自动重启服务
4. 回滚后自动验证 `/api/health`

交付物：

- 回滚脚本
- 回滚操作说明

---

### 阶段 E：目标机验证

目标：

- 在目标 Windows 电脑上完成真实演练

任务：

1. 安装 Node.js LTS
2. 安装 Git
3. 拉取 GitHub 仓库
4. 配置 `.env`
5. 迁移数据库到独立 `data` 目录
6. 首次启动验证
7. 执行一次更新脚本演练
8. 执行一次回滚演练

交付物：

- 目标机运行确认
- 更新演练记录
- 回滚演练记录

---

## 4. 详细任务清单

### A1. 仓库治理

- 明确主仓库地址
- 明确默认分支
- 明确是否使用 tag/release
- 确认 `.gitignore` 不包含真实数据库

### A2. 运行目录治理

- 确认 Windows 根目录
- 确认 `app/data/backups/logs` 目录职责
- 明确哪些文件允许本地修改

### B1. 路径配置化

- 增加 `DB_PATH`
- 增加 `SESSION_DB_DIR`
- 检查启动逻辑
- 检查初始化逻辑
- 检查脚本默认路径

### B2. 环境模板治理

- 增加生产环境示例 `.env`
- 文档化配置项
- 明确默认端口与密钥规则

### C1. 更新脚本

- 停止服务
- 备份数据库
- 更新代码
- 安装依赖
- 重启服务
- 健康检查

### C2. 服务兼容

- 确认 Windows 服务模式
- 确认手动启动模式
- 明确两套更新路径是否一致

### D1. 回滚脚本

- 选择上个版本
- 恢复数据库
- 重启服务
- 输出结果

### E1. 实机验证

- 首次部署验证
- 更新验证
- 回滚验证

---

## 5. 实施顺序建议

建议严格按下面顺序执行：

1. 目录与仓库规范
2. 数据路径配置化
3. 更新脚本
4. 回滚脚本
5. Windows 实机演练

不要跳过第 2 步直接做更新脚本，否则容易把数据库继续绑死在代码目录里。

---

## 6. 风险点

### 风险 1：数据库仍然放在代码目录

影响：

- 更新代码时容易误覆盖数据

控制方式：

- 必须先完成路径配置化

### 风险 2：better-sqlite3 在目标机安装失败

影响：

- 服务无法启动

控制方式：

- 目标机先验证 `npm install`

### 风险 3：更新脚本成功但服务未真正可用

影响：

- 以为更新成功，实际页面打不开

控制方式：

- 更新后必须检查 `/api/health`

### 风险 4：没有回滚演练

影响：

- 出现问题时无法快速恢复

控制方式：

- 上线前必须完成一次真实回滚演练

---

建议每个任务按以下状态推进：

- `未开始`
- `进行中`
- `已完成`
- `阻塞`

建议跟踪字段：

- 任务编号
- 任务名称
- 负责人
- 状态
- 风险
- 备注

---

## 8. 完成标准

当以下项目全部完成时，可认为该方案开发实施完成：

1. GitHub 仓库方案明确
2. Windows 目录结构明确
3. 数据路径可配置
4. 更新脚本可执行
5. 回滚脚本可执行
6. 目标机启动成功
7. 目标机更新演练成功
8. 目标机回滚演练成功
