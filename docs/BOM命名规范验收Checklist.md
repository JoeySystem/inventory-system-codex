# BOM命名规范验收 Checklist

## 一、验收目的

本 Checklist 用于验证系统是否已经具备承接并执行 BOM 命名规范的能力。

本验收特别强调：

- 新 BOM 应严格执行新规范
- 旧 BOM 应支持渐进过渡
- 不允许用“一刀切强改”代替治理

---

## 二、验收范围

验收覆盖：

- 数据模型
- BOM 新建/编辑流程
- 命名规则校验
- 治理工作台
- 旧 BOM 过渡机制
- 搜索与版本兼容
- 引用与历史解释安全性

---

## 三、阶段性验收

## A. 阶段 1：规则打底

### 数据字段

- [ ] `boms` 已增加 `bom_level`

- [ ] `boms` 已增加 `suggested_name`
- [ ] `boms` 已增加 `legacy_name`

### 新建 BOM

- [ ] 新建 BOM 必须能填写显示版本
- [ ] 名称可按结构化方式生成

- [ ] 新建 BOM 不允许继续使用 `.BOM / 模板 / 最终版 / 日期入名`

### 编辑旧 BOM

- [ ] 旧 BOM 仍可打开和编辑
- [ ] 旧 BOM 不会因为历史命名不合规而被直接锁死

## 四、命名规则验收

### 基础格式

- [ ] BOM 名称统一使用 `_` 分隔
- [ ] 名称至少包含 4 段
- [ ] 第一段为 BOM 层级
- [ ] 最后一段为 `VNN`

### 层级规则

- [ ] 整机以 `整机_` 开头
- [ ] 模块以 `模块_` 开头
- [ ] 板级以 `板级_` 开头
- [ ] 配套件以 `配套件_` 开头
- [ ] 名称层级与 `bom_level` 一致

### 禁用项

- [ ] 名称中不再允许日期
- [ ] 名称中不再允许 `.BOM`
- [ ] 名称中不再允许 `模板`
- [ ] 名称中不再允许 `最新版`
- [ ] 名称中不再允许 `最终版`

### 版本规则

- [ ] 显示版本统一为 `V01 / V02 / V03`
- [ ] 名称中的版本段与 `display_version` 一致
- [ ] 不再依赖日期或“新版/最终版”表达版本

---

## 五、旧 BOM 过渡机制验收

### 过渡原则

- [ ] 历史 BOM 不会被一次性强改
- [ ] 历史 BOM 不会因不合规而统一禁止使用
- [ ] 历史 BOM 有明确的 `legacy_exempt` 或等价状态
- [ ] 新建 BOM 与旧 BOM 的规则处理方式有明确区别

### 业务安全

- [ ] 改名不会破坏历史工单快照
- [ ] 改名不会破坏 BOM 版本历史
- [ ] 改名不会破坏 BOM 被引用关系

### 可见性

- [ ] 用户可识别旧 BOM 是否属于历史豁免
- [ ] 用户可识别哪些旧 BOM 需要优先整改

## 八、建议的验收通过标准

建议分两档判断：

### 基础通过

满足以下即可判定“系统已具备承接能力”：

- [ ] 新建 BOM 已强校验
- [ ] 旧 BOM 有识别与提示
- [ ] 有命名治理工作台
- [ ] 至少 10 个试点 BOM 已整改
- [ ] 历史 BOM 有豁免机制

### 完整通过

满足以下即可判定“系统已完成规范落地”：

- [ ] 活跃 BOM 大部分完成规范化
- [ ] 历史豁免边界清晰
- [ ] 用户已按新规则稳定使用
- [ ] 命名治理进入日常维护
