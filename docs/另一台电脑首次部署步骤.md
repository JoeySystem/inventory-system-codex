# 另一台电脑首次部署步骤

本文用于指导另一台 Windows 电脑从 GitHub 首次拉取并部署 `OvO-Inventory-System`。

名称说明：

- `OvO-Inventory-System` 指 GitHub 仓库名，也是后续统一使用的项目名称

## 1. 前置准备

目标电脑需要先安装：

- Git
- Node.js 20 LTS、22 LTS 或 24.x
- GitHub Desktop 或可用的 Git 凭证

当前支持 Node `20.x / 22.x / 24.x`。  
正式运行机仍优先推荐 `20.x LTS` 或 `22.x LTS`，以保持长期更新稳定性。

建议准备固定目录，例如：

- `D:\OvO-System\app`：代码目录
- `D:\OvO-System\data`：数据库目录
- `D:\OvO-System\backups`：备份目录

## 2. 首次拉取代码

打开终端或 Git Bash，执行：

```bash
cd /d D:\OvO-System
git clone https://github.com/JoeySystem/OvO-Inventory-System.git app
cd app
```

如果使用 SSH：

```bash
git clone git@github.com:JoeySystem/OvO-Inventory-System.git app
cd app
```

## 3. 安装依赖

在代码目录执行：

```bash
npm install
npm run check-env
npm run preflight
```

首次安装时如果 `better-sqlite3` 编译失败，需要先确认：

- Node 版本是否为 LTS
- 系统是否安装了必要的编译组件

其中：

- `check-env` 用于检查 Node / npm 版本与原生依赖兼容性
- `preflight` 用于检查数据库路径、session 目录和前端本地 vendor 资源是否齐备

## 4. 准备环境文件

复制环境模板：

```bash
copy .env.example .env
```

将 `.env` 修改为目标电脑路径，建议至少配置：

```env
PORT=3000
NODE_ENV=production
COOKIE_SECURE=auto
TRUST_PROXY=0
SESSION_SECRET=请替换为新的随机字符串
DB_PATH=D:\OvO-System\data\inventory.db
SESSION_DB_DIR=D:\OvO-System\data
```

可选：

```env
SESSION_DB_NAME=sessions.db
```

说明：

- 本地 `http://localhost` 调试时，建议保持 `NODE_ENV=production`
- 登录态由 `COOKIE_SECURE=auto` 控制
- 不要通过把 `NODE_ENV` 改成 `development` 来规避 cookie 问题
- `SESSION_SECRET` 必须替换为强随机字符串；若仍使用占位值，生产环境会拒绝启动

## 5. 放置数据库

### 情况 A：继续使用当前真实业务数据

将当前运行中的数据库文件复制到目标电脑：

- `inventory.db`
- 如需保留登录会话，可一起复制 `sessions.db`

放到：

- `D:\OvO-System\data\inventory.db`
- `D:\OvO-System\data\sessions.db`

### 情况 B：全新初始化

如果暂时不迁真实数据，可由系统初始化空库。

## 6. 首次启动验证

方式一：直接运行 Node

```bash
npm start
```

方式二：使用部署脚本

```bat
deploy\install.bat
deploy\start.bat
```

## 7. 健康检查

启动后检查：

- 浏览器打开 `http://localhost:3000`
- 健康检查接口：`http://localhost:3000/api/health`

应确认：

- 页面可打开
- 登录正常
- 物料、BOM、生产、报表能进入
- 数据库路径读取正确

## 8. 首次部署验收

至少验证以下内容：

1. 能正常登录
2. 能看到当前业务数据
3. `收 / 发 / 转 / 盘` 页面能打开
4. `生产 / 报表 / 一致性治理` 页面能打开
5. `/api/health` 返回成功

## 9. 后续更新方式

首次部署完成后，后续更新优先使用：

```bat
deploy\update.bat
```

或：

```bat
deploy\update-and-restart.bat
```

如果更新失败，可使用：

```bat
deploy\rollback.bat <tag或commit> <数据库备份文件名>
```

## 10. 注意事项

- 不要把真实 `.env` 上传到 GitHub
- 不要把真实数据库重新提交进仓库
- 代码更新与数据目录要分开管理
- 正式机建议只更新经过验证的 tag，不建议长期直接拉未验证提交
