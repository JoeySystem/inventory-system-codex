# GitHub部署与Windows自更新方案

## 1. 文档目的

本文档用于明确当前物料管理系统后续在另一台 Windows 电脑上的部署方式、更新方式、数据隔离方式，以及后续项目跟进时的实施路径。

目标不是“把项目传到另一台电脑上能跑”，而是建立一套长期可维护的机制：

- 代码可以持续更新
- 数据不会被代码更新覆盖
- 更新过程可以回滚
- 后续新功能和 Bug 修复可以稳定下发

---

## 2. 推荐总方案

推荐采用以下整体结构：

1. GitHub 仓库负责管理代码
2. Windows 运行机负责保存真实业务数据
3. 数据库与代码目录分离
4. 通过更新脚本执行：
   - 备份
   - 拉取代码
   - 安装依赖
   - 运行迁移
   - 重启服务

即：

- **代码层可更新**
- **数据层不被覆盖**
- **运行层可回滚**

---

## 3. 为什么不建议直接整目录覆盖

如果把整套项目直接放在 GitHub 上，然后在 Windows 机器上直接 `git pull` 覆盖整个运行目录，会有几个明显风险：

1. 真实数据库可能被误覆盖
2. `.env` 可能被开发配置污染
3. 备份文件和运行态文件会混在代码目录里
4. 回滚困难
5. 后续多次更新后，运行目录会逐渐失控

所以推荐方案不是“整目录覆盖更新”，而是：

- **代码单独更新**
- **数据单独保存**

---

## 4. 推荐目录结构

在 Windows 目标机器上建议固定一个根目录，例如：

```text
D:\OvO-System\
```

推荐结构：

```text
D:\OvO-System\
├─ app\                 # GitHub 拉下来的代码
├─ data\                # 真实数据库与运行数据
├─ backups\             # 更新前数据库备份
├─ logs\                # 服务日志
└─ .env                 # 本地环境配置
```

说明：

- `app/` 可以反复更新
- `data/` 不参与 Git 更新
- `.env` 放在根目录或 `app/` 外部，避免更新时被覆盖

---

## 5. 代码与数据分离原则

### 5.1 代码目录

代码目录建议只放：

- 前端文件
- 后端代码
- 路由
- 服务
- 迁移脚本
- 部署脚本

### 5.2 数据目录

数据目录建议保存：

- `inventory.db`
- `sessions.db`
- 后续导入文件缓存（如需要）
- 本地备份文件

### 5.3 环境配置

环境配置建议至少包括：

- `PORT`
- `NODE_ENV`
- `SESSION_SECRET`
- `DB_PATH`
- `SESSION_DB_DIR`

后续应让服务通过环境变量读取数据库路径，而不是长期固定使用项目目录下的 `data/inventory.db`。

---

## 6. 推荐 GitHub 使用方式

### 6.1 仓库用途

GitHub 仓库负责：

- 保存代码
- 保存部署脚本
- 保存迁移脚本
- 保存文档

GitHub 仓库不负责：

- 保存真实生产数据库
- 保存本地运行日志
- 保存本地账号会话

### 6.2 分支策略建议

建议至少使用：

- `main`
  - 开发主线
- `release/*`
  - 预发布或正式发布版本

如果流程更简单，也可以：

- 日常开发合并到 `main`
- Windows 运行机只更新打过版本 tag 的提交

### 6.3 发布建议

建议正式运行机器不要直接盯着随时变化的主分支，而是：

- 使用 `Git tag`
- 或使用 GitHub Release

这样可以明确：

- 当前机器运行的是哪个版本
- 出问题时可以回滚到哪个版本

---

## 7. 更新模式对比

### 模式 A：直接 `git pull`

适用：

- 内部测试机
- 开发联调机

优点：

- 简单
- 更新快

缺点：

- 风险高
- 可能拉到半成品代码
- 不适合作为长期正式运行方案

### 模式 B：按 Tag / Release 更新

适用：

- 正式运行机器
- 公司内部真实业务使用环境

优点：

- 可控
- 可回滚
- 更适合稳定维护

缺点：

- 多一步版本发布动作

### 建议结论

正式运行机推荐使用：

- **Tag / Release 更新**

不推荐长期直接追踪 `main`。

---

## 8. Windows 机器上的更新流程

建议后续每次更新都走统一脚本，不靠人工逐步敲命令。

标准更新流程应为：

1. 停止当前服务
2. 备份数据库
3. 拉取指定版本代码
4. 安装依赖
5. 执行数据库迁移
6. 启动服务
7. 验证健康检查

---

## 9. 推荐新增的部署脚本

建议后续新增：

### 9.1 `deploy/update.bat`

职责：

- 拉取代码
- 安装依赖
- 运行迁移

### 9.2 `deploy/update-and-restart.bat`

职责：

- 停服务
- 备份数据库
- 执行更新
- 重启服务
- 健康检查

### 9.3 `deploy/rollback.bat`

职责：

- 回退到上一个版本
- 恢复数据库备份
- 重启服务

---

## 10. 更新脚本建议包含的动作

### 10.1 备份数据库

更新前必须先做：

- 备份 `inventory.db`
- 备份 `sessions.db`（可选）

建议命名格式：

```text
backups\inventory-YYYYMMDD-HHMMSS.db
```

### 10.2 拉取代码

两种候选方式：

- 更新到指定 tag
- 更新到指定分支最新提交

正式机建议：

- 默认更新到指定 tag

### 10.3 安装依赖

建议：

```bash
npm install --production
```

### 10.4 运行迁移

项目当前是启动时自动迁移，但更新脚本里也应明确执行或至少显式验证。

### 10.5 服务重启

如果用了 Windows 服务：

- 停止服务
- 更新
- 启动服务

如果是手动运行：

- 关闭旧窗口
- 重新执行 `start.bat`

### 10.6 健康检查

更新后至少检查：

- `http://localhost:3000`
- `http://localhost:3000/api/health`

## 11. 数据库路径改造建议

这是整套方案里最关键的一步。

当前项目默认数据库在：

```text
data/inventory.db
```

为了支持 GitHub 更新而不覆盖数据，建议改成：

- `DB_PATH` 可配置
- `SESSION_DB_DIR` 可配置

例如：

```env
DB_PATH=D:\OvO-System\data\inventory.db
SESSION_DB_DIR=D:\OvO-System\data
```

这样无论 `app/` 怎么更新，数据库都不会跟着移动或被覆盖。

### 需要改造的代码点

主要涉及：

- `server/db/database.js`
- `server/index.js`
- `server/db/init.js`
- 可能涉及个别脚本中的数据库默认路径

---

## 12. 正式部署时的推荐环境

Windows 11 机器建议：

- Node.js 20 LTS 或 22 LTS
- Git
- 稳定网络
- SSD 硬盘
- 至少 16GB 内存

并建议：

- 安装目录不要放在 iCloud、OneDrive 等同步目录里
- 使用固定盘符目录
- 保证 Windows 用户对目录有读写权限

## 13. 更新失败时的回滚方案

必须事先定义回滚规则。

### 13.1 回滚触发条件

出现以下任意情况时回滚：

- 服务启动失败
- 登录失败
- 核心页面打不开
- `/api/health` 不通过
- 数据迁移异常

### 13.2 回滚动作

1. 停止新版本服务
2. 切回上一个代码版本
3. 恢复更新前数据库备份
4. 重启服务
5. 再次验证健康检查

### 13.3 回滚前提

要做到可回滚，必须提前保留：

- 上一个 Git tag / 版本号
- 更新前数据库备份

### 第一阶段：结构准备

目标：

- 确认 GitHub 仓库可作为代码源
- 确认 Windows 机器运行目录结构
- 明确代码目录与数据目录分离

产出：

- Windows 目标目录规范
- `.env` 规范
- GitHub 仓库使用规则

目标：

- 支持数据库路径可配置
- 支持 session 路径可配置

产出：

- 数据目录与代码目录解耦

### 第三阶段：更新脚本

目标：

- 增加一键更新脚本
- 增加更新前自动备份
- 增加基础健康检查

产出：

- `update.bat`
- `update-and-restart.bat`

### 第四阶段：上线与验证

目标：

- 在目标 Windows 机器试运行
- 完整执行一次更新流程
- 完整执行一次回滚流程

产出：

- 更新可行性确认
- 回滚可行性确认

## 16. 验收标准

满足以下条件时，可以认为该方案完成可用：

1. 代码可以从 GitHub 拉到 Windows 机器
2. 数据库不与代码目录强绑定
3. 更新不会覆盖真实业务数据
4. 更新脚本能一键完成代码更新
5. 更新后服务可正常启动
6. `/api/health` 正常
7. 更新失败时可以回滚
8. 至少完成一次真实更新演练

---

## 17. 当前结论

当前项目已经具备以下基础：

- 标准 Node 服务入口
- SQLite 本地数据库
- Windows 启动与安装脚本
- 适合放到 GitHub 管理代码

但要达到“另一台 Windows 机器下载后，后续可以自己稳定更新”的目标，还需要继续补齐：

- 数据路径可配置
- 代码与数据目录分离
- 一键更新脚本
- 回滚机制

也就是说：

- **GitHub 管代码：现在就可以做**
- **Windows 稳定自更新：需要按本文档继续实施**
