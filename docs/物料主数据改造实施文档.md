# 物料主数据改造实施文档

## 1. 文档目的

本文档用于指导 `inventory-system-codex` 项目完成“物料主数据中心”改造，目标是将当前偏台账型的物料模块升级为可支撑库存、生产、发货、报表闭环的主数据系统。

本文档面向：

- 产品设计
- 前后端开发
- 数据迁移实施
- 测试与上线验收

## 2. 改造目标

本次改造完成后，系统应具备以下能力：

- 物料具备统一、稳定、可治理的主身份
- 物料可明确区分原材料、半成品、成品、辅料、包材等角色
- 物料可挂接采购、生产、销售、库存控制属性
- 物料与 BOM、SOP、生产工单建立清晰边界和快照关系
- 库存控制从单一 `min_stock` 升级为策略化模型
- 物料生命周期、替代关系、单位换算、审计日志可支撑后续扩展
- 所有关键业务模块围绕统一物料主数据工作

## 3. 当前问题摘要

当前版本的物料模块存在以下结构性问题：

- 物料字段过少，无法承接采购、生产、销售、库存控制
- 只有简单分类，没有分类驱动的默认业务规则
- 没有物料类型，成品、半成品、原料都混在同一模型里
- 只有单单位模型，无法支持采购单位、库存单位、发货单位换算
- 没有生命周期状态，只有启停用
- 没有物料替代关系、版本治理、查重治理
- 生产模块对 BOM/SOP 的引用关系不稳定，无法形成闭环
- 当前库存预警仅依赖 `min_stock`，口径过于粗糙

## 4. 实施范围

本次实施包含以下范围：

- 数据库表结构升级
- 后端 API 扩展与契约重构
- 前端物料模块重构
- 与 BOM / SOP / 生产 / 仓库 / 发货模块的关联改造
- 存量数据迁移与清洗
- 上线验收口径补齐

本次实施暂不包含：

- 完整采购模块
- 财务总账或应收应付模块
- 高级审批流引擎

## 5. 目标业务模型

### 5.1 物料主档

物料主档应成为系统统一数据根，承担以下职责：

- 提供唯一物料身份
- 提供业务使用属性
- 提供库存控制策略
- 提供生产与销售可用性定义
- 提供分析与治理基础

### 5.2 业务边界

- `materials`：物料主档根实体
- `categories`：分类树及分类默认规则
- `material_uoms`：单位与换算
- `material_suppliers`：默认供应商及供货关系
- `material_substitutions`：替代料关系
- `material_snapshots`：关键主数据快照
- `boms`：定义“做什么料要用什么料”
- `sops`：定义“怎么做”
- `production_orders`：引用 BOM/SOP 快照执行业务

## 6. 数据模型设计

### 6.1 现有 `materials` 表扩展字段

## 7. 数据库迁移方案

### 7.1 迁移原则

- 不直接覆盖历史主数据
- 先加字段，再回填，再切换业务读写逻辑
- 迁移期间保留旧字段兼容
- 所有结构迁移可重复执行

### 7.2 迁移步骤

1. 新增表与新增字段
2. 建立默认分类树与分类默认规则
3. 对现有物料按分类批量回填 `material_type`
4. 回填经营属性
5. 回填库存控制字段
6. 生成 `spec_key`
7. 为所有物料补基础单位记录
8. 回填 `lifecycle_status`
9. 生成数据质量标记
10. 上线兼容版 API
11. 前端切到新字段
12. 清理旧口径依赖

### 7.3 存量数据回填规则

建议初始回填规则如下：

- 分类名含“成品” => `finished`
- 分类名含“半成品” => `wip`
- 分类名含“BOM清单” => `raw`
- 分类名含“螺丝/辅料/胶/包装” => `consumable` 或 `packaging`
- 分类名无法判断 => `raw`，并打 `data_quality_status = incomplete`

经营属性默认规则：

- `finished`：`is_sellable=1`, `is_producible=1`
- `wip`：`is_producible=1`
- `raw`：`is_purchasable=1`
- `consumable`：`is_purchasable=1`

生命周期默认规则：

- `is_active=1` => `active`
- `is_active=0` => `inactive`

### 7.4 重复物料识别规则

初步规则：

- 同 `name + spec + brand + unit`
- 同 `code` 但不同 `id`
- 同 `spec_key` 且分类一致

命中后：

- 不自动合并
- 标记为 `duplicate_suspected`
- 进入治理清单

## 8. 后端 API 设计

### 8.1 物料主档接口

#### `GET /api/materials`

新增筛选参数：

- `materialType`
- `lifecycleStatus`
- `isPurchasable`
- `isProducible`
- `isSellable`
- `negativeStockOnly`
- `lowStockOnly`
- `dirtyOnly`
- `categoryId`

#### `GET /api/materials/:id`

返回结构应扩展为：

- 基本信息
- 经营属性
- 库存控制
- 成本信息
- 单位换算
- 供应商关系
- 替代料
- 关联 BOM
- 关联 SOP
- 最近库存流水
- 审计日志

#### `POST /api/materials`

新增输入字段：

- `materialType`
- `lifecycleStatus`
- `businessFlags`
- `stockPolicy`
- `costPolicy`
- `uoms`
- `suppliers`
- `substitutions`

校验规则：

- 主编码唯一
- `materialType` 必填
- 至少存在一个基础单位
- 成品必须允许销售或生产之一
- 若 `defaultBomId` 存在，物料必须可生产

#### `PUT /api/materials/:id`

新增行为：

- 支持乐观锁字段 `version_no`
- 关键字段变更时写入 `material_snapshots`
- 生命周期变更写入 `material_lifecycle_logs`

### 8.2 生命周期接口

#### `PUT /api/materials/:id/lifecycle`

输入：

- `toStatus`
- `reason`

状态机建议：

- `draft -> pending_review -> active`
- `active -> frozen`
- `frozen -> active`
- `active/frozen -> inactive`
- `inactive -> obsolete`

约束：

- `draft` 不允许进入 BOM / SOP / 工单 / 发货
- `frozen` 不允许新业务引用
- `obsolete` 不允许恢复

### 8.3 主数据治理接口

#### `GET /api/materials/duplicates`

返回疑似重复物料分组。

#### `POST /api/materials/:id/merge`

输入：

- `targetMaterialId`
- `reason`

行为：

- 合并前检查是否允许合并
- 迁移所有可迁移外键引用
- 写入 `material_merge_logs`
- 旧物料置为 `obsolete`

#### `GET /api/materials/:id/usages`

返回引用关系：

- BOM
- SOP
- 生产工单
- 发货单
- 库存记录
- 流水记录

### 8.4 单位换算接口

#### `PUT /api/materials/:id/uoms`

输入：

- `baseUnit`
- `purchaseUnit`
- `salesUnit`
- `productionUnit`
- `conversions`

约束：

- 比例必须大于 0
- 基础单位不可删除

### 8.5 替代料接口

#### `GET /api/materials/:id/substitutions`

#### `POST /api/materials/:id/substitutions`

#### `DELETE /api/materials/:id/substitutions/:subId`

### 8.6 导入导出接口

#### 新增导入流程

`POST /api/materials/import/preview`

- 只校验，不落库
- 返回新增、更新、冲突、重复、非法分类、非法单位

`POST /api/materials/import/commit`

- 提交 preview token
- 具备幂等性
- 全量成功或整批失败

导出应支持：

- 基础主档
- 扩展属性
- 库存控制参数
- 单位换算
- 供应商关系

## 9. 与其他模块的改造关系

### 9.1 与 BOM 的关系

- BOM 只能引用 `active` 状态物料
- BOM 项必须引用物料快照
- 成品类物料可绑定默认 BOM
- 替代料规则可用于 BOM 试算，但不自动落地

### 9.2 与 SOP 的关系

- SOP 只定义工艺步骤和步骤用料
- SOP 不承担物料主身份
- SOP 中引用物料时也应带快照

### 9.3 与生产工单的关系

工单创建时必须冻结以下内容：

- 工单引用 BOM 版本
- 工单引用 SOP 版本
- 工单用料快照
- 工单产出物料快照
- 工单单位换算快照

新增表建议：

- `production_order_materials`
- `production_order_outputs`
- `production_order_scrap`

### 9.4 与库存模块的关系

库存模块读取物料主数据中的：

- 基础单位
- 是否允许负库存
- 是否批次管理
- 是否序列号管理
- 安全库存与补货点

### 9.5 与发货模块的关系

发货模块读取物料主数据中的：

- 销售可用性
- 销售单位
- 售价
- 替代编码

## 13. 验收标准

### 13.1 主数据验收

- 新建物料必须选择物料类型
- 物料至少有一个基础单位
- 生命周期状态切换符合状态机
- 冻结或停用物料不能被新单引用

### 13.2 业务闭环验收

- 成品物料可绑定默认 BOM / SOP
- 工单创建后即冻结物料快照
- 修改物料主档不会影响历史工单解释
- 发货只能引用可销售物料
- 原材料只能在允许场景下被领料或出库

### 13.3 数据治理验收

- 重复物料可被识别
- 导入前可预览冲突
- 合并物料后历史单据引用仍可追溯
- 负库存异常可从物料视角快速查看

### 13.4 报表验收

- 物料详情页能展示库存、流水、BOM、SOP、单据历史
- 低库存规则基于新库存策略生效
- 负库存异常报表可按分类与类型聚合

## 15. 风险与注意事项

- 当前系统是单文件前端，物料页复杂度提升后可维护性会快速下降
- 若不先做前后端结构拆分，后续功能迭代成本会越来越高
- 若不同时重构 BOM / SOP / 生产引用方式，物料主数据升级价值会被削弱
- 存量数据质量一般，建议先做一轮数据诊断再批量回填

## 16. 建议实施顺序

建议按以下顺序推进：

1. 先完成数据库扩展和迁移脚本
2. 再完成后端 API 重构
3. 再完成前端物料模块重构
4. 最后打通 BOM / SOP / 生产 / 发货联动
