# Git 标签发布与更新流程规范

本文用于统一 `inventory-system-codex` 的版本发布、更新、回滚流程。

## 1. 目标

建立一套可重复、可回滚、适合 Windows 运行机执行的发布规范：

- 开发改动先在本地完成并验证
- 提交代码到 GitHub
- 通过 `tag` 标识可部署版本
- Windows 运行机只更新到明确版本
- 出问题时可快速回滚

## 2. 分支与版本约定

默认使用：

- 主分支：`main`
- 发布标签：`vX.Y.Z`

例如：

- `v1.0.0`
- `v1.0.1`
- `v1.1.0`

建议规则：

- 修复 bug：补丁版本 `Z + 1`
- 小功能新增：次版本 `Y + 1`
- 结构性变化：主版本 `X + 1`

## 3. 发布前检查

发布 tag 前至少完成：

1. 本地代码提交干净
2. 系统能启动
3. 关键页面可访问
4. 健康检查通过
5. 数据库迁移已验证
6. 本次改动已写清楚说明

建议命令：

```bash
git status
git log --oneline --max-count=5
```

## 4. 标准发布流程

### 第一步：提交代码

```bash
git add .
git commit -m "描述本次改动"
git push origin main
```

### 第二步：打 tag

```bash
git tag -a v1.0.0 -m "v1.0.0"
git push origin v1.0.0
```

### 第三步：在 GitHub 创建 Release

建议在 GitHub 仓库中：

- 选择已推送的 tag
- 填写版本说明
- 记录本次主要改动、风险、回滚点

## 5. Windows 运行机更新规范

不建议长期直接更新到 `main`。  
正式机建议优先更新到明确 tag。

推荐方式：

```bat
deploy\update-and-restart.bat v1.0.0
```

如果只是测试机，可临时更新当前分支：

```bat
deploy\update-and-restart.bat
```

## 6. 回滚规范

若新版本有问题：

1. 先确认回滚目标版本
2. 找到更新前备份的数据库
3. 执行回滚脚本

示例：

```bat
deploy\rollback.bat v1.0.0 inventory_2026-03-26_103000.db
```

回滚后应再次验证：

- `http://localhost:3000`
- `http://localhost:3000/api/health`

## 7. 推荐发布节奏

建议不要每修一个小问题就让正式机更新。

更稳的做法：

- 开发期：本地频繁提交
- 阶段性稳定后：打 tag
- 正式机只更新已验证 tag

## 8. 版本说明建议模板

每次 tag / release 建议记录：

- 版本号
- 发布时间
- 主要功能变化
- 数据库迁移情况
- 是否需要备份
- 是否需要人工校验
- 已知风险

## 9. 适合当前项目的推荐做法

对本项目建议执行：

1. 所有开发先提交到 `main`
2. 本地验证通过后打 tag
3. Windows 正式机只更新 tag
4. 更新前自动备份数据库
5. 更新失败用 `rollback.bat` 回滚

## 10. 当前仓库状态建议

当前仓库已经适合继续按该规范运作，但有两类文件后续建议谨慎管理：

- `dist/*.zip`
  - 属于发布产物，后续可继续保留，也可改由 GitHub Release 附件承载
- `.claude/launch.json`
  - 属于本地开发配置，后续若与同事开发环境无关，可视情况从仓库移除

## 11. 验收标准

当以下条件满足时，可认为发布与更新流程已经规范化：

1. 可以打 tag 并推送
2. Windows 运行机可以按 tag 更新
3. 更新前会自动备份数据库
4. 更新后会自动做健康检查
5. 回滚脚本能恢复到指定版本和数据库备份
