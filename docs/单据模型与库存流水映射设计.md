# 单据模型与库存流水映射设计

## 1. 文档目的

本文档用于定义 `inventory-system-codex` 项目在“收 / 发 / 转 / 盘”新架构下的单据模型、库存流水模型、状态机规则与报表映射关系。

本文档解决的问题包括：

- `收 / 发 / 转 / 盘` 在系统中对应哪些单据
- 单据如何驱动库存余额变化
- 库存流水需要记录哪些关键字段
- 如何保证 `期初 + 入 - 出 = 期末`
- `转` 与 `盘` 在报表中如何处理
- 生产工单、发货单、仓库操作如何统一进入库存流水

本文档面向：

- 后端开发
- 数据库设计
- 报表开发
- 测试
- 产品设计

## 2. 设计目标

本次设计完成后，系统应满足以下目标：

- 前台可按 `收 / 发 / 转 / 盘` 组织业务操作
- 底层库存变动全部统一为标准流水
- 单据与流水一一可追踪
- 所有影响库存的业务都有明确来源类型
- 转与盘不会污染经营层面的进销存口径
- 报表可以稳定输出数量和金额口径

## 3. 总体技术原则

### 3.1 单据驱动原则

库存余额不能直接改写，必须由单据执行后驱动。

允许触发库存变化的对象：

- 收货单
- 发料/发货单
- 调拨单
- 盘点/调整单
- 生产工单派生动作

### 3.2 双层分类原则

库存流水必须同时记录：

- 动作类型 `movement_type`
- 业务来源类型 `biz_type`

说明：

- `movement_type` 用于表达库存方向
- `biz_type` 用于表达业务原因

### 3.3 来源可追溯原则

每一条库存流水都必须能追溯到：

- 来源单据类型
- 来源单据 ID
- 操作人
- 执行时间

### 3.4 报表统一口径原则

所有报表必须由库存流水汇总产生，不允许：

- 页面现算一套
- 单据现算一套
- 余额表单独写一套

## 4. 目标单据体系

## 4.1 总体单据分层

建议形成两层单据体系：

### A. 业务主单据

代表业务动作的正式单据：

- 收货单 `receipts`
- 发出单 `issues`
- 调拨单 `transfers`
- 盘点单 `stock_counts`
- 调整单 `stock_adjustments`

### B. 业务来源单据

代表上游业务背景的来源对象：

- 采购收货单
- 销售发货单
- 生产工单
- 退料单
- 报废单
- 客退单

说明：

短期内可以不立刻拆出完整新表，但必须在现有表中补足语义字段，最终达到同样效果。

## 4.2 与当前系统兼容的建议

当前系统已有：

- 发货单
- 仓库出入库动作
- 生产工单

建议兼容路径如下：

### 短期

继续沿用现有表，但统一补：

- `biz_type`
- `source_doc_type`
- `source_doc_id`

### 中期

逐步引入标准动作单据：

- `receipts`
- `issues`
- `transfers`
- `stock_counts`
- `stock_adjustments`

### 长期

上游业务单据与动作单据分层：

- 工单负责控制
- 动作单据负责落库
- 流水负责核算

## 5. 单据类型设计

## 5.1 收单据

### 业务定义

表示库存增加的正式执行单据。

### 建议 `biz_type`

- `purchase_receipt`
- `production_receipt`
- `return_to_stock`
- `customer_return`
- `outsource_return`
- `manual_receipt`

### 主要字段

- 单号
- 来源类型
- 来源单据 ID
- 收货仓库
- 日期
- 状态
- 明细行
- 审核信息
- 备注

### 对库存的影响

- 增加目标仓库存
- 生成 `movement_type = in`

## 5.2 发单据

### 业务定义

表示库存减少的正式执行单据。

### 建议 `biz_type`

- `sales_shipment`
- `production_issue`
- `sample_issue`
- `scrap_issue`
- `manual_issue`

### 主要字段

- 单号
- 来源类型
- 来源单据 ID
- 发出仓库
- 日期
- 状态
- 领用/客户对象
- 明细行
- 审核信息
- 备注

### 对库存的影响

- 扣减发出仓库存
- 生成 `movement_type = out`

## 5.3 转单据

### 业务定义

表示库存位置变化但总量不变的执行单据。

### 建议 `biz_type`

- `warehouse_transfer`
- `workshop_transfer`
- `line_side_replenishment`
- `return_from_line_side`

### 主要字段

- 单号
- 转出仓
- 转入仓
- 日期
- 状态
- 明细行
- 接收确认信息
- 备注

### 对库存的影响

- 转出仓减少
- 转入仓增加
- 流水层通常形成两笔：
  - 一笔 `out`
  - 一笔 `in`

### 报表口径

- 不计入经营层面的“入”和“出”
- 仅计入位置变化分析

## 5.4 盘单据

### 业务定义

表示库存核对和差异修正单据。

### 拆分建议

- 盘点单 `stock_counts`
- 调整单 `stock_adjustments`

### 建议 `biz_type`

- `count_gain`
- `count_loss`
- `manual_adjust`
- `scrap_adjust`

### 对库存的影响

- 根据差异方向增加或减少库存
- 生成 `movement_type = count_adjust`

### 报表口径

- 盘盈视为“入”
- 盘亏视为“出”
- 但需在差异分析报表中单列

## 6. 库存流水模型设计

## 6.1 目标表设计

建议以现有 `stock_movements` 为基础，逐步升级为统一库存流水表。

关键字段建议如下：

- `id`
- `movement_type`
  - 枚举：`in`, `out`, `transfer`, `count_adjust`
- `biz_type`
  - 枚举见后文
- `source_doc_type`
- `source_doc_id`
- `source_line_id`
- `material_id`
- `warehouse_id`
- `to_warehouse_id`
- `quantity`
- `unit_name`
- `unit_cost`
- `total_amount`
- `direction_sign`
  - 入为 `+1`
  - 出为 `-1`
- `occurred_at`
- `created_by`
- `approved_by`
- `status`
- `notes`

## 6.2 字段含义

### movement_type

描述库存动作方向：

- `in`
- `out`
- `transfer`
- `count_adjust`

### biz_type

描述业务来源原因：

- `purchase_receipt`
- `production_receipt`
- `return_to_stock`
- `customer_return`
- `production_issue`
- `sales_shipment`
- `sample_issue`
- `scrap_issue`
- `warehouse_transfer`
- `count_gain`
- `count_loss`
- `manual_adjust`

### source_doc_type

记录来源单据类型，例如：

- `shipment`
- `production_order`
- `warehouse_receipt`
- `warehouse_issue`
- `transfer`
- `count`
- `adjustment`

### source_doc_id

记录来源单据主键。

### direction_sign

用于报表快速聚合：

- 入库为 `+1`
- 出库为 `-1`

## 6.3 转单据的流水实现建议

调拨建议生成两条流水：

1. 转出流水
   - `movement_type = transfer`
   - `direction_sign = -1`
2. 转入流水
   - `movement_type = transfer`
   - `direction_sign = +1`

但在报表聚合时：

- 不将 `biz_type = warehouse_transfer` 计入经营入/出

## 7. 单据状态机设计

## 7.1 通用状态

所有动作单据建议统一状态：

- `draft`
- `pending`
- `approved`
- `executed`
- `cancelled`

### 状态含义

- `draft`
  - 可编辑，不影响库存
- `pending`
  - 待审核，不影响库存
- `approved`
  - 已批准，待执行
- `executed`
  - 已正式影响库存
- `cancelled`
  - 已取消，不再执行

## 7.2 库存影响时点

统一要求：

- 只有 `executed` 状态才能影响库存

不得出现：

- 建单即扣库存
- 审核前已落库存
- 取消后不回滚

## 7.3 幂等要求

所有执行动作必须具备幂等保护。

建议：

- 单据执行时生成唯一执行号
- 重试时识别已执行流水
- 禁止同一单据重复落库存

## 8. 报表映射规则

## 8.1 核算主公式

报表层使用统一公式：

- 期末 = 期初 + 本期入 - 本期出

## 8.2 入库口径

以下 `biz_type` 计入“入”：

- `purchase_receipt`
- `production_receipt`
- `return_to_stock`
- `customer_return`
- `count_gain`
- `manual_adjust` 中的正向调整

## 8.3 出库口径

以下 `biz_type` 计入“出”：

- `sales_shipment`
- `production_issue`
- `sample_issue`
- `scrap_issue`
- `count_loss`
- `manual_adjust` 中的负向调整

## 8.4 不计入经营入/出的口径

以下类型不应进入经营层面的入/出汇总：

- `warehouse_transfer`
- `workshop_transfer`
- 其他纯位置移动类动作

## 8.5 盘点口径

盘点差异不应被隐藏在普通入出中，而应：

- 在进销存报表中进入入/出
- 在盘点差异报表中单列展示

## 9. 与 BOM / SOP / 生产管理的映射

## 9.1 BOM

BOM 不直接生成库存流水。

作用：

- 定义生产物料结构
- 作为工单领料的理论依据

## 9.2 SOP

SOP 不直接生成库存流水。

作用：

- 定义工艺步骤
- 定义步骤级物料使用关系

## 9.3 生产工单

生产工单是最重要的业务控制单据之一。

### 工单与流水的关系

#### 工单开始

- 触发 `发单据`
- `biz_type = production_issue`

#### 工单完工

- 触发 `收单据`
- `biz_type = production_receipt`

#### 工单取消

- 若已领料，触发：
  - `收单据` 或 `调整单`

#### 工单异常

- 触发：
  - `盘单据`
  - `报废调整`

### 技术结论

- 工单本身不是库存流水
- 工单负责驱动动作单据和库存流水

## 10. 与现有模块的兼容设计

## 10.1 发货管理

现有发货单可继续作为来源单据：

- `source_doc_type = shipment`
- `biz_type = sales_shipment`

短期无需删除原表，只需补齐映射。

## 10.2 仓库出入库

现有仓库手工入出库可映射为：

- 手工入库 -> `manual_receipt`
- 手工出库 -> `manual_issue`
- 手工调拨 -> `warehouse_transfer`

## 10.3 生产管理

现有生产工单可映射为：

- 领料 -> `production_issue`
- 完工 -> `production_receipt`
- 取消退料 -> `return_to_stock`

## 11. 数据库改造建议

## 11.1 第一阶段：现有表补字段

建议先在现有流水或相关表上补充：

- `biz_type`
- `source_doc_type`
- `source_doc_id`
- `executed_at`
- `approved_by`

## 11.2 第二阶段：统一库存流水写入服务

新增统一服务层，例如：

- `inventoryMovementService`

负责：

- 单据执行
- 流水写入
- 余额更新
- 幂等校验
- 日志记录

## 11.3 第三阶段：统一报表查询口径

所有报表查询统一基于库存流水生成。

## 12. 服务层设计建议

建议新增统一服务接口：

### 12.1 executeReceipt

输入：

- 收货单
- 明细
- 来源信息

输出：

- 入库流水
- 更新后的库存余额

### 12.2 executeIssue

输入：

- 发出单
- 明细
- 来源信息

输出：

- 出库流水
- 更新后的库存余额

### 12.3 executeTransfer

输入：

- 调拨单
- 明细

输出：

- 转出流水
- 转入流水
- 更新后的仓库库存

### 12.4 executeCountAdjustment

输入：

- 盘点差异单

输出：

- 调整流水
- 更新后的库存余额

## 13. 测试与验收要点

## 13.1 单据执行测试

必须覆盖：

- 草稿不落库存
- 审核不落库存
- 执行后落库存
- 取消后不重复落库存
- 重试不重复落库存

## 13.2 报表口径测试

必须验证：

- 收单据进入“入”
- 发单据进入“出”
- 转单据不进入经营入/出
- 盘盈盘亏正确进入差异与进销存

## 13.3 工单联动测试

必须验证：

- 开工触发领料
- 完工触发入库
- 取消触发退料或调整

## 14. 风险点

### 14.1 常见错误

- 用页面模块名替代 `biz_type`
- 调拨只写一条流水
- 盘点差异不进报表
- 工单直接改库存而不生成动作流水
- 重试导致重复落库

### 14.2 控制措施

- 强制所有库存变化通过统一服务
- 强制每条流水带来源单据
- 报表只读统一流水
- 对执行动作加幂等键

## 15. 结论

新架构下的技术落地关键不在于菜单，而在于：

- 单据模型是否清楚
- 库存流水是否统一
- 状态机是否合法
- 报表口径是否稳定

本设计的核心结论是：

- 前台使用 `收 / 发 / 转 / 盘`
- 中间通过 `biz_type` 和来源单据建立业务语义
- 后台通过统一库存流水实现 `期初 + 入 - 出 = 期末`

只有这样，系统才能同时满足：

- 仓库员好用
- 车间主管好懂
- 财务报表能对
- 老板看数能解释
